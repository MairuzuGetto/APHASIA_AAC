## Decisions（今日决策）

### DEC-01

Document Guideline 与 Product Blueprint 分离。
Document Guideline 仅负责文档规范。
不负责产品设计。

### DEC-02

Design Tokens 不预先填写数值。
所有尺寸、字体、间距、

统一等待：
Flutter Prototype
完成后再回填。

Design Tokens 定义为：
**Living Specification**

### DEC-03

Component Guide 不负责 Design。

Component Guide 负责：
> 定义所有 Component。

Design System 负责：
> 定义设计原则。

Flutter Architecture 负责：
> 定义程式架构。

职责正式分离。

### DEC-04

Component Guide 采用统一 Template。
未来所有 Component
皆使用同一份模板建立。
保持一致性。

### DEC-05

Component Library 采用功能分类。

分类如下：
- Communication
- Navigation
- Feedback
- Input
- Settings
- Foundation

而非依页面分类。

### DEC-06

目前不拆分多个 Markdown。

维持：
> 03_Component_Guide.md

待未来 Component 数量增加后，再考虑拆分。
避免过度设计。

## Lessons Learned（今日收获）

今天最大的收获：

**不要为了未来可能发生的事情，增加今天的复杂度。**

例如：

- 不预先拆 Component 文件
- 不预先填写 Token 数值
- 不预先决定 Radius、Padding

设计规格先建立，实现参数后验证。

## Reflection（每日回顾）

今天是一个重要的里程碑。

前几天，我们一直在讨论产品理念与设计原则；今天，我们正式建立了 **Component Guide** 的整体框架。

更重要的是，我们逐渐形成了一套一致的开发流程：

    Document Guideline
            ↓
    Product Blueprint
            ↓
    Design System
            ↓
    Component Guide
            ↓
    Flutter Architecture
            ↓
    Flutter Source Code

今天也确认了一个贯穿整个项目的重要原则：

> **先定义规格（Specification），再验证实现（Prototype）；先建立原则，再确定参数**。

这意味着未来无论是 Design Tokens、组件尺寸还是动画细节，都将以原型验证为依据，而不是凭经验预设。
